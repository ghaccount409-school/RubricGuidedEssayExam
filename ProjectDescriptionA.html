OVERVIEW

Modern educational systems are under extreme stress because it is virtually impossible now to give, and grade, written work and guarantee that it has not been written by AI. The same is true for many types of incremental homework.

Because of this, assessment and standards are now moving back toward sit-down [hand-written] exams. But this places a very large grading burden on teachers and professors, and can be problematic for online and especially online-asynchronous courses. It also degrades the educational experience when all the focus is on big exams.

In addition, there are further constraints for asynchronous (online) classes: If students cannot be required to take an exam on the same day at the same time, then exams will always potentially "leak" when students, say, take the exam on a Monday, and then tell their friends what is on the exam so they can unfairly prepare in a targeted way to take the same exam later in the week.

But AI technology, which has created the problem, can also be used to help solve it, working around constraints.

One of the best kinds of exams is an oral exam, given one on one with a professor who simply asks questions of the student and listens to the answers. Previously, for general classes this is impractical because of the numbers of students a professor has each quarter.

But following is a design that can mimic this kind of exam, requiring only a web browser (possibly a locked-down browser), a proctoring environment, and a hybrid program that accesses an LLM in real time. This is what we will build in various forms this quarter.

Furthermore, with hugely expanded testing capabilities that do not burden professors, along with the rise of personal AI tutors, this opens the door to modular course design, wherein students complete education modules at their own pace and are accurately tested on what they have learned when they complete a module. Modular education design and pacing can greatly improve the educational experience. In the past, general adoption of such modular education designs has been blocked because of the high-level of personal interaction needed by each student.

The design for the exposition below uses Python, but any modern programming language that can interpret code in real time will serve.

DESIGN

Our goal is to create a testing and exam-question evaluation system that generates essay-style test questions in real time, and evaluates essay style written answers provided by students taking the exam.

In this project we use (primarily) English-language prompts to write code that writes English-language prompts that direct an LLM to generate code in real time, which is passed back to Python and executed there.

Prompt Template System

Using AI tools, we prepare a mostly-completed template for a prompt that will be sent to an LLM (through together.ai) in real time during Python program execution.

Under program control in Python, we use string manipulation to complete the template to create a full prompt. The prompt has a sample Python data object that will be completed, and filled with data, by the LLM, then returned to Python.

LLM Instructions for Question Generation

The prompt instructs the LLM to do the following in the chosen domain:

  1. To create each essay question and the associated grading rubric, read any information provided explicitly to the LLM by the professor, and also gather domain information trained into the LLM's general knowledge base in that domain area.
  2. Create an information sheet in the domain that may be displayed to the student as background information for the exam question.
  3. Create an essay exam question based on the material, and related to the displayed information. Design a grading rubric at the same time that will be paired with the question. Specifically, what information should be present in the essay answer the student provides?
  4. For each question, return the background display information, the essay question, the grading rubric, and anything else necessary (such as specific domain information that the student might be expected to know for this question), inside the constraints of the Python data object, to Python as a response from the LLM.

Student Testing Interface

Then Python, through a FastAPI / uvicorn server (or other) presents a (series of) web page(s) to the student taking the test, asking the series of essay questions.

The student test-taker types her responses to the questions into the provided input boxes in essay format and this information is passed back to Python over the Web.

Grading Process

For each question answered, the Python program prepares another prompt containing the question, the associated grading rubric, all the saved background information, and the student's essay response, possibly along with other information such as how long the student spent on the question.

The "student response prompt" is sent to the LLM, with instructions (in the prompt) to grade the student response along several dimensions, using the associated rubric and saved background information. An explanation is also required, giving the details of why the student response was graded the way it was.

If the student has given a highly satisfactory answer, all of the grading information is placed back in a different Python data structure and returned to Python. We will call this state P.

Extension for a More Sophisticated Program

Otherwise, if the answer is not of high quality, in a more sophisticated version of the design, if the student has not given a fully satisfactory response, the LLM prepares a follow-up question and grading rubric to see if something can be done to improve the situation. For example, the LLM might present a question such as "Can you please expand on the statement 'Blah blah blah' to make sure it applies to [fill in the blank]."

This is returned to Python, that displays the follow-up to the student.

This is possibly repeated several times, and the LLM uses ALL of the information gathered to grade the question and the follow-ups. All of the information along with the grade assessment for the question (and the explanation for how the grade was determined) is placed in a Python data object and returned to Python. Now we are at state P for a student that has had follow-ups.

Database Storage & Final Grading

From state P, the graded question, the grading rubric for the questions, the saved background information, and the explanation of how the question was graded is placed in a database table of "graded exam questions" indexed by the student's unique ID for storage.

When all of the exam questions are answered and stored, Python selects particular grading summary information from each question given and passes the entire collection to the LLM. The LLM assesses a final total grade, and then returns the total grade, along with an explanation for how it was derived, back to Python inside of a Python data object. There are many variations on how to compile the final grade.

Advanced: Curved Grading

Advanced projects would look at a collection of all responses by ALL students for similar questions (note the difficulty of determining "similar"), stored in the database and use a partial-order sorting process (from best response to worst response) to add curved grading to the class when giving an overall exam grade. This can also include all previous exam responses in the same domain from previously archived exams. Professors often set standards based not only on the current students, but also on the work of many previous students over the years.

Prompt Construction & AI Collaboration

Obviously, there will be a great deal of Prompt construction in the Python server program, preparing everything to be sent to the LLM in various iterations, but the intention of this project is that you use AI coding tools, possibly even exclusively instructing your AI coding collaborator in English, such that the AI collaborator writes all of the actual prompt templates for you, that appear as constants (or as prompt template files to be read in at run time) inside of the Python program.

In this way you are using English to create prompts that instructs a coding LLM (Cursor / Claude Code) to create prompts that instructs a different LLM (together.ai) to create executable Python code. Along the way your prompts also instruct your AI coding collaborator to write a lot of test-manipulation Python code, and all the JSON for the interface, as well.

The buzz-word for this is "Vibe Coding" (as of February 2025) but I prefer my own phrase of using an AI Collaborator, which I have been using since 2023.

⚠️ WARNING

Executing Python code that was written in real time by an LLM can be dangerous. For this project the Python code is primarily an assignment of a Python object to a variable. But in the larger world, and for anything fancy you would want to run the realtime LLM-created code inside a sandbox that protects your own Python code from executing foreign code that has full access to your system.

Other Notes

Variation: Discussion-Based Testing

In a variation of this exam design, your very sophisticated system can engage in a discussion with the student regarding the domain in question, testing the student's knowledge in a different manner. This would be a variation of the mentor/tutor design we will have seen elsewhere. Does the student ask appropriate questions? Do they understand new concepts that require extensive knowledge in the domain. For example, if the examiner / tutor engages in a dialog about programming for dialogs with LLMs, and the student asks intelligent questions about stateless programming, this would suggest they have good knowledge in that area.

For this design, in an interesting twist, you would need to keep a history of the conversation and pass a version of it back to the LLM to review each time, because, in fact, like the web, LLMs operate in a stateless way, with each new connection starting from scratch. (When AI "remembers" where you are in conversations with it, it is because each time you enter new text, it reviews the entire conversation you have been holding, which it has saved externally to the LLM intelligence in the interface software.)

Grading: Novel Assessment Approaches

Given the power of LLMs, their extensive domain knowledge and the free-form answers provided by students, there are many, many exciting and novel ways to go about assessing grades. While, ultimately we will want a "percentage correct" final grade (e.g. 93%) we can use all sorts of different quantitative and qualitative metrics to compile it. Give this aspect some thought, and possibly allow some selection of different schemes by the professor. For example, some may prefer more creative answers, and others may prefer that students only say things that stick strictly to the facts.

When the exam is over, your system may display the grade back to the student along with explanations of how the grading was determined.

Dispute System

You might want a way for a student to dispute any particular grading of a question. In this case you would store their complaint/explanation along with an AI assessment of the validity of the complaint for later review by the professor. Or, in a particularly cool extension, the AI could regrade on the spot if the argument made was valid. In general, however, you would want this to take place later.

If you invite students to dispute answers immediately, you will be overloaded with emotional, typically non-logical responses. If you wait until some time has passed, disputes are more likely to be valid, with cool heads prevailing.

You should provide a way for the complete exam, along with grades, rubrics and grading explanations to be easily reviewable over the web interface by the professor.

Most Challenging Aspect

The most challenging aspect of this project will be in specifying to the LLM how to constrain the domain knowledge to be tested to exactly the right level of specificity. The display screen before each question can help with this. For this reason, a special project role of Educational / Prompt Design Specialist will be required, and this work will be central in determining your success.

It is possible that your Python program could pass the PowerPoint lecture slides for the class to the LLM to help in preparing essay questions.

Input into Your Program

1. You will need at least the following:

  • Professor input as domain information background for creating the actual questions, and possibly as help on the creating the grading rubrics. But note, we don't really want this to be specific questions. Instead we want the LLM to generate the questions in real time (because we don't want exactly the same questions given) based on very specific areas of knowledge, and do the grading, while the test is being taken.
  • Answers from the students while taking the exam.

2. You may also want the following:

  • Possible dispute arguments for when a student feels their answer was not graded correctly.
  • Domain choices input by the professor for constraining the area of each of the exam questions.
  • Weighting information for grading.
  • Time limits for each test question.
  • Exam time limit or other alterations for students with learning or physical disabilities.
  • Etc.
